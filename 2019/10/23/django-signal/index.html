    filefield를 통해 오디오 파일을 업로드 한 후 폼이 저장될 때 오디오의 재생 시간을 함께 저장하고 싶었다. 만약에 업로드할 파일을 선택한 뒤 filefiled가 사용자의 로컬에 있는 주소를 기억한다면, 로컬에서 바로 처리를 한 후 파일과 함께 업로드 될 순 없을까?에서 출발한 질문은 아무리 찾아봐도 답을 얻기는 어려웠다. 어디선가 사용자 파일에 접근하는 것은 보안상의 이유로 알 수 없다고 어렴풋이 본 것도 같다.

    그게 안되면 업로드 할 파일을 우선 업로드한 뒤 s3의 url를 통해 추가로 입력해 준다면 어떨까? 라는 생각이 들었는데, db에 두번 접근을 하는게 영 찜찜해 하고 있는 동안 django signal에 대해 알게 되었다.

    다음은 django signal에 대한 공식문서를 구글번역기의 힘을 빌어보았다.

    Django Signal

    Django에는 “신호 분배기(signal dispatcher)”가 포함되어있는데, 프레임 워크의 다른 곳에서 작업이 발생할 때 분리 된 응용 프로그램에 알릴 수 있습니다. 간단히 말해서, 시그널은 특정 발신자가 일련의 어떤 동작이 일어났음을 수신자에게 알릴 수 있습니다. 코드의 많은 부분이 동일한 이벤트에 관심이있을 때 특히 유용합니다.

    Django 자체에서 특정 동작을 사용자 코드에 알릴 수 있는 내장 코드 세트를 제공합니다. 여기에는 몇 가지 유용한 알림이 포함됩니다.

    • django.db.models.signals.pre_save & django.db.models.signals.post_save

      모델의 save()메소드가 호출 되기 전이나 후에 전송 됩니다.

    • django.db.models.signals.pre_delete & django.db.models.signals.post_delete

      모델의 delete() 메소드 또는 queryset의 delete() 메소드가 호출 되기 전이나 후에 전송 됩니다.

    • django.db.models.signals.m2m_changed

      ManyToManyField모델이 변경 될 때 전송됩니다 .

    • django.core.signals.request_started & django.core.signals.request_finished

      Django가 HTTP 요청을 시작하거나 마치면 전송됩니다.

    전체 목록과 각 시그널에 대한 자세한 설명은 빌트 시그널 문서를 참조하십시오.

    사용자 정의 시그널을 정의하고 보낼 수도 있습니다. 아래 내용을 참조하십시오.

    신호 듣기 ¶

    시그널을 받으려면 Signal.connect() 메서드를 이용해 리시버 함수를 등록해야된다. 시그널을 보내면 리시버 함수가 호출된다.

    Signal.connect( receiver , sender = None , weak = True , dispatch_uid = None )

    • 매개 변수 :
      • receiver - 이 시그널에 연결될 콜백 기능. 자세한 내용은 리시버 함수를 참조하십시오.
      • sender – 시그널을 수신할 특정 발신자를 지정합니다. 자세한 내용은 특정 발신자가 보낸 시그널 연결을 참조하십시오 .
      • weak – Django는 기본적으로 신호 처리기를 약한 참조로 저장합니다. 따라서 리시버가 지역함수인 경우 가비지 컬렉션 될 수 있습니다. 이를 방지하려면 시그널의 connect()메소드 를 호출 할 때 weak=False로 패스 하십시오 .
      • dispatch_uid – 중복 시그널이 전송될 수 있는 경우 신호 수신기의 고유 식별자입니다. 자세한 내용은 중복 시그널 방지 를 참조하십시오.
        각 HTTP 요청이 완료될 때 마다 호출할 신호를 등록하여 이것이 어떻게 동작하는지 봅시다. request_finished 시그널에 연결하겠습니다.

    리시 기능버 ¶

    먼저 리시버 함수를 정의해야합니다. 리시버는 파이썬 함수 또는 메소드 일 수 있습니다.

    def my_callback(sender, **kwargs):
    print("Request finished!")

    함수는 sender, 와일드 카드 키워드 인수(**kwargs)를 인수로 취합니다. 모든 시그널 핸들러는 이 인수를 사용해야합니다.

    잠시후에 발신자를 보겠지만, 지금은 **kwargs 인수를 살펴 봅니다. 모든 시그널은 키워드 인수를 전송하며 언제든지 키워드 인수를 변경할 수 있습니다. request_finished에 관해서는 인수를 보내지 않는 것으로 문서화되어 있다. 그말인 즉슨 신호 처리를 다음과 같이 my_callback(sender) 쓰려는 유혹을받을 수 있습니다.

    이것은 틀렸을 것입니다 – 사실, 그렇게하면 장고는 오류를 던질 것입니다. 이유는 어떤 시점에서든 인수가 신호에 추가 될 수 있고 수신자가 새로운 인수를 처리 할 수 ​​있어야하기 때문입니다.

    ####리시버 함수 연결하기 ¶
    수신기를 신호에 연결할 수있는 두 가지 방법이 있습니다. 수동 연결 경로를 사용할 수 있습니다.

    from django.core.signals import request_finished

request_finished.connect(my_callback)

    또는 receiver()데코레이터를 사용할 수 있습니다 .

    receiver 시그널)

    • 매개 변수 :
      • signal – 함수를 연결할 신호 또는 신호 목록.

    데코레이터와 연결하는 방법은 다음과 같습니다.

    from django.core.signals import request_finished
from django.dispatch import receiver

@receiver(request_finished)
def my_callback(sender, **kwargs):
    print("Request finished!")

    이제 my_callback요청이 완료 될 때마다 함수가 호출됩니다.

    이 코드는 어디에 살아야합니까?

    엄밀히 말하면 신호 처리 및 등록 코드는 원하는 곳 어디에서나 사용할 수 있지만 models코드 가져 오기의 부작용을 최소화하기 위해 응용 프로그램의 루트 모듈과 해당 모듈을 피하는 것이 좋습니다 .

    실제로 신호 처리기는 일반적으로 시그널 관련 응용 프로그램의 하위 모듈에 정의 됩니다. 신호 수신기는 어플리케이션 환경설정 클래스의 ready()매서드에 연결됩니다. receiver() 데코레이터를 사용하면 signals 하위 모듈 내부의 ready()를 간단하게 가져올 수 있다. ready().

    노트

    ready() 메서드는 테스트 기간 동안 한 번 이상 실행될 수 있기에 중복으로 부터 시그널을 보호하는 것이 좋습니다. 특히, 테스트 내에서 시그널을 보내려는 경우 더욱 좋습니다.

    ####특정 발신자가 보낸 신호에 연결 ¶
    몇몇 시그널은 여러 번 전송되지만 일부의 시그널을 수신하는 데 관심이 있습니다. 예를 들어, 모델이 저장되기 전에 전송 된 django.db.models.signals.pre_save 신호를 생각해보세요 . 대부분 구체적인 하나의 모델이 아닌 모델이 저장될 때마다 알 필요는 없습니다.

    이 경우 특정 발신자가 보낸 시그널만 수신하도록 등록 할 수 있습니다. django.db.models.signals.pre_save의 경우 보낸 사람이 저장되는 모델 클래스가되므로 일부 모델에서만 시그널을 전송하도록 지정할 수 있습니다.

    from django.db.models.signals import pre_save
from django.dispatch import receiver
from myapp.models import MyModel


@receiver(pre_save, sender=MyModel)
def my_handler(sender, **kwargs):
...

    이 my_handler함수는 MyModel의 인스턴스가 저장 될 때만 호출됩니다 .

    신호마다 발신자마다 다른 객체가 사용되므로, 각 특정 시그널에 대한 자세한 내용은 내장 신호 문서를 참조해야 합니다.

    ####중복 시그널 방지
    상황에 따라 수신기를 시그널에 연결하는 코드가 여러 번 실행될 수 있습니다. 이로 인해 수신기 함수가 한 번 이상 등록되어 단일 신호 이벤트에 대해 여러 번 호출 될 수 있습니다.

    이 동작에 문제가있는 경우 (예 : 시그널을 사용하여 모델을 저장할 때마다 이메일을 보낼 때) 고유 식별자를 dispatch_uid인수 로 전달 하여 수신기 함수를 식별하십시오. 이 식별자는 일반적으로 문자열이지만 해시 가능한 객체이면 충분합니다. 결과적으로 수신기 함수는 고유한 dispatch_uid값을 한번만 받을 것입니다.

    from django.core.signals import request_finished

request_finished.connect(my_callback, dispatch_uid="my_unique_identifier")

    ####신호 정의 및 전송
    애플리케이션은 신호 인프라를 활용하여 자체 시그널을 제공 할 수 있습니다.

    커스텀 시그널을 사용하는 경우

    시그널은 내포된 함수 호출로 디버깅을 어렵게 만듭니다. 커스텀 시그의 발신자와 수신자가 모두 프로젝트 내에있는 경우 명시적인 함수 호출을 사용하는 것이 좋습니다.

    시그널 정의와 전송

    class Signal(providing_args=list) [source]

    모든 시그널은 django.dispatch.Signal 인스턴스입니다. providing_args는 리스너에게 줄 시그널의 인수 이름 목록입니다. 그러나 시그널이 실제로 수신자에게 이러한 인수를 주는지 확인할 수 있는 것은 아무것도 없기 때문에 이것은 순전히 문서상 얘기입니다

    예를 들면 다음과 같습니다.

    import django.dispatch

pizza_done = django.dispatch.Signal(providing_args=["toppings", "size"])

    이것은 toppings, size 인수를 리시버에게 넘겨줄 pizza_done 시그널을 선언합니다.

    언제든지이 인수 목록을 변경할 수 있으므로 첫 번째 시도에서 API를 가져 올 필요없습니다.

    ####시그널 보내기
    장고에서 시그널을 보내는 방법은 두 가지가 있습니다.

    Signal.send( 발신자 , ** kwargs ) [source]

    Signal.send_robust( 발신자 , ** kwargs ) [source]

    신호를 보내려면 Signal.send()(내장 시그널이 모두 사용) 또는 Signal.send_robust()을 호출하십시오. sender인수(대부분의 클래스)를 제공해야하 원하는 만큼 다른 키워드 인수를 제공 할 수 있습니다.

    예를 들어 pizza_done 시그널을 보내는 방법 은 다음과 같습니다.

    class PizzaStore:
...

    def send_pizza(self, toppings, size):
        pizza_done.send(sender=self.__class__, toppings=toppings, size=size)
        ...

    send()와 send_robust() 모두 호출된 리시버 함수와 응답 값 목록을 튜플 쌍의 리스트로 [(receiver, response), … ]을 반환한다.

    send()는 처리된 리시버 함수에 의해 발생 되는 예외가 send_robust()와 다릅니다. send()는 리시버에 의해 발생한 예외를 포착 하지 않습니다; 단순히 오류를 전파합니다. 따라서 오류가 발생했을 때 모든 리시버에 시그널이 표시되는 것은 아닙니다.

    send_robust()는 Python Exception 클래스 에서 파생 된 모든 오류를 포착하고 모든 리시버에게 시그널을 알립니다. 오류가 발생하면 오류가 발생한 리시의 튜플 쌍으로 오류 인스턴스가 리턴됩니다.

    send_robust() 호출시 반환 된 오류의 속성을 traceback에서 역추적 할 수 있습니다.

    ####신호 분리 ¶
    Signal.disconnect( receiver = None , sender = None , dispatch_uid = None ) [소스] ¶
    시그널로 부터 리시버의 연결을 끊을려면 Signal.disconnect()를 호출하십시오 . 인수는 Signal.connect()에 설명되어 있습니다. 연결이 끊어졌으면 True, 아니라면 False 메소드를 반환 합니다 .

    receiver인수는 연결이 끊어진 리시버를 나타냅니다. None경우 dispatch_uid리시버를 식별하는 데 사용됩니다.
